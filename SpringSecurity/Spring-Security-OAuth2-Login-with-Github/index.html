Spring Security - OAuth2 Login with Github

Register Github App

Login to Github, go to Settings/Developer settings/OAuth Apps. And then create OAuth App

Register App

  • Set Homepage URL to be http://localhost:8080
  • Set Authorization callback URL to be http://localhost:8080/login/oauth2/code/github

After The app is register, you should have a Client ID and Client Secret.

Follow Tutorial Here https://spring.io/guides/tutorials/spring-boot-oauth2/

Login Process

Before login, the CLient application needs to register itself with the Auth Server. After registration, you will get Client id and Client secret.

Client Registration
client_id csu7XlYm3GcFHqk7U5T8Gg3d
client_secret Wf8aKSajDZXoRgEUjNT3Ms3f7PdKD2tlu3ZtndsSefbrP9Yv
User Account
login nervous-mosquito@example.com
password Innocent-Orangutan-21

  1. Build The authorization URL and redirect the user to the authorization server
  2. After the user is redirected back to the client, verify the state matches
  3. Exchange the authorization code for an access token

Users are redirected to authorization server

http://localhost:8080/login will redirect to Github’s login page

Redirect to OAuth server’s login page

The user can now login and authorize the application.

Client receive Authorization Code

The user was redirected back to the client, and you’ll notice a few additional query parameters in the URL:

?state=CBdV_KAU2mLu0V6S&code=lh19cPvRnjQ-q4j1JvZDIs1zDePV_f9HOYLwm0ZjssXByutq

You need to first verify that the state parameter matches the value stored in this user’s session so that you protect against CSRF attacks.

Depending on how you’ve stored the state parameter (in a cookie, session, or some other way), verify that it matches the state that you originally included in step 1. Previously, we had stored the state in a cookie for this demo.

Does the state stored by the client (CBdV_KAU2mLu0V6S) match the state in the redirect (CBdV_KAU2mLu0V6S)?

Client exchange Authorization Code

Now you’re ready to exchange the authorization code for an access token.

The client builds a POST request to the token endpoint with the following parameters:

POST https://authorization-server.com/token

grant_type=authorization_code
&client_id=csu7XlYm3GcFHqk7U5T8Gg3d
&client_secret=Wf8aKSajDZXoRgEUjNT3Ms3f7PdKD2tlu3ZtndsSefbrP9Yv
&redirect_uri=https://www.oauth.com/playground/authorization-code.html
&code=lh19cPvRnjQ-q4j1JvZDIs1zDePV_f9HOYLwm0ZjssXByutq

Note that the client’s credentials are included in the POST body in this example. Other authorization servers may require that the credentials are sent as a HTTP Basic Authentication header.

Authorization Server send the access token and refresh token to Client

Token Endpoint Response

Here’s the response from the token endpoint! The response includes the access token and refresh token.

{
“token_type”: “Bearer”,
“expires_in”: 86400,
“access_token”: “GEEPLhQucIL6feeb6D21nRXijg_uRxuPB0owlLx0Om7AiMpVQJVQ1mLyEb6lZ4kPO8DZ37fh”,
“scope”: “photo offline_access”,
“refresh_token”: “wiE0J8IXZKoXffGKcLll7LQL”
}
Great! Now your application has an access token, and can use it to make API requests on behalf of the user.
